# 核心概念总览

来源：

- https://solana.com/zh/docs
- https://solana.com/zh/docs/core

## 这部分讲什么

Solana 的核心开发模型可以简化成下面几层：

1. `账户`：链上状态都放在账户里。
2. `程序`：程序本身通常无状态，负责读取和修改账户。
3. `指令`：调用某个程序的一次请求。
4. `交易`：一个或多个指令的原子打包。
5. `费用`：发送交易必须付费，还可以加优先费。
6. `PDA`：程序可确定性推导出的地址，没有私钥。
7. `CPI`：程序在执行中调用别的程序。

## 建议学习顺序

推荐按这个顺序读：

1. 账户
2. 程序
3. 指令
4. 交易
5. 费用
6. PDA
7. CPI

原因很简单：

- 程序操作的是账户
- 指令是调用程序的最小单位
- 交易负责把多条指令一起提交
- PDA 和 CPI 都建立在前面这些概念之上

## 一张脑图式理解

```text
用户 / 客户端
  -> 构造指令
  -> 把指令打包成交易
  -> 签名并发送
  -> Validator 执行程序
  -> 程序读取/修改账户
```

## 一个最小例子

假设你要“转 1 SOL”：

1. 你需要两个账户：发送方账户、接收方账户
2. 你要调用 `System Program`
3. 你要构造一条“转账指令”
4. 你把这条指令放进交易
5. 发送方签名
6. 网络执行后，发送方余额减少，接收方余额增加

## 开发时最常见的误区

- 误区 1：以为程序里可以像传统后端一样直接保存状态  
  事实：状态在账户里，不在程序二进制里。

- 误区 2：以为一条交易只能做一件事  
  事实：一笔交易可以包含多条指令。

- 误区 3：以为 PDA 有私钥  
  事实：PDA 没有私钥，它的“签名”来自程序通过 `invoke_signed` 证明控制权。

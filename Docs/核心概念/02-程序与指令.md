# 程序与指令

来源：

- https://solana.com/zh/docs/core/programs/index
- https://solana.com/zh/docs/core/instructions/index

## 程序是什么

Solana 程序本质上是一个可执行账户，里面存的是 sBPF 字节码。

它有两个关键特征：

- `executable = true`
- 通常本身不保存可变状态

所以最重要的理解是：

- 程序负责“规则”
- 账户负责“状态”

## 指令是什么

指令是“调用程序做某件事”的请求。  
一条指令通常包含三部分：

- `program_id`：要调用哪个程序
- `accounts`：这次执行需要哪些账户
- `data`：程序要处理的参数字节

## 一条指令的思维模型

```text
请调用某个程序
并给它这些账户
再把这些参数交给它
```

## 例子：转账指令

用户转账 1 SOL，本质上是在调用 `System Program` 的转账逻辑。

这条指令大概表达的是：

```text
program_id = System Program
accounts = [发送方, 接收方]
data = { amount: 1 SOL }
```

## 例子：留言板写入指令

假设你有一个 `post_message` 指令：

```text
program_id = 留言板程序
accounts = [message_account, author]
data = { instruction: post_message, content: "hello" }
```

程序收到后会：

1. 检查作者是否签名
2. 检查 `message_account` 是否可写
3. 把 `"hello"` 写进账户数据

## 为什么一条指令只属于一个程序

官方文档明确指出，一条指令只针对一个程序。  
如果你要同时调用多个程序，要么：

- 在交易里放多条指令
- 或者在程序内部做 CPI

## 程序执行时的几个限制

从官方概念页可以抓住几个重要边界：

- 程序编译为 sBPF
- 堆和栈都有大小限制
- 调用深度有限
- 执行会消耗计算单元（CU）

这意味着：

- 不要把程序写成“超大循环”
- 不要在链上做复杂重计算
- 能预先在客户端算的，尽量放客户端

## 开发时常见误区

- 误区 1：把 `data` 当成可读字符串  
  实际上它通常是程序自定义的字节序列，需要序列化/反序列化。

- 误区 2：指令里不传账户，程序也能访问  
  不行。程序只能访问当前传入的账户。

- 误区 3：程序账户保存业务状态  
  不对。业务状态通常要放数据账户。

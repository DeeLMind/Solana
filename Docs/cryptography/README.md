# Solana 加密算法与密码学基础

## 1. 为什么要看加密算法

Solana 不是“靠某一种密码算法运行”的链，而是组合了多种密码学工具：

- 公私钥签名
- 哈希函数
- 地址派生
- 状态完整性校验

理解这些基础后，你再看账户、交易、PDA、PoH、Merkle 证明会顺很多。

## 2. Ed25519 签名

Solana 最核心的签名算法是 `Ed25519`。

它主要用于：

- 钱包签名交易
- 验证账户授权
- 节点身份与消息认证

### 它为什么重要

因为在 Solana 里，绝大多数“谁有权限做这件事”的判断，最终都要落回：

- 这个交易是否带有合法签名
- 这个 signer 是否和预期公钥一致

### 你在开发里会经常看到的东西

- `Keypair`
- `Pubkey`
- `Signer`

这些概念都建立在 Ed25519 公私钥体系上。

## 3. SHA-256

`SHA-256` 在 Solana 中非常常见。

它用于：

- PoH 连续哈希
- 某些地址派生与种子处理
- 数据摘要和完整性校验

PoH 本质上就强依赖连续哈希序列，因此哈希函数不是边角料，而是系统时间序的基础之一。

## 4. PDA 的密码学本质

`PDA` 是 Program Derived Address。

它的本质不是普通钱包地址，而是：

- 通过种子 + program id 推导出来的地址
- 这个地址没有对应的普通私钥
- 只能由对应程序通过 `invoke_signed` 等机制代表它“签名”

### 为什么这很重要

因为 PDA 让程序可以安全地拥有状态和权限，而不用真的保管私钥。

这是一种非常关键的链上权限设计。

## 5. 哈希与地址派生

Solana 里很多“确定性地址”的概念都离不开哈希：

- PDA derivation
- Address seeds
- 某些索引键设计

开发者最该理解的不是内部每个字节细节，而是：

- 地址不是总要随机生成
- 很多地址可以根据规则稳定推导
- 可推导地址有助于减少链上状态查找成本

## 6. Merkle Tree 与状态证明

虽然 Solana 的普通开发流程里，你不会像写钱包签名那样天天碰 Merkle Tree，但它仍然是非常重要的基础结构。

常见用途包括：

- 数据完整性验证
- 压缩状态相关方案
- 轻量证明结构

如果你后面研究压缩 NFT、状态证明、bridge 或 offchain indexing，就会越来越常见。

## 7. 随机数问题

链上程序通常不能简单把“当前环境里的某个值”当成安全随机数。

在 Solana 中，随机性设计是敏感问题，因为：

- 链上执行需要可重放和可验证
- 节点不能各自得到不同执行结果
- 可预测“伪随机”很容易被利用

所以所有涉及抽奖、游戏、排序的逻辑，都要非常谨慎处理随机性来源。

## 8. 密钥、助记词与钱包

用户平时最常接触的是：

- 助记词
- 私钥文件
- keypair
- 钱包地址

需要区分：

- 助记词是便于人类备份的入口
- 私钥是签名能力本身
- 公钥是公开身份
- 地址通常就是公钥表示

## 9. 开发与审计最关心的密码学风险

### 签名检查错误

最常见的问题之一不是算法本身被攻破，而是程序忘了检查：

- 该账户是不是 signer
- signer 是否就是预期身份

### PDA 种子设计不严谨

如果种子空间设计混乱，可能带来：

- 地址冲突风险
- 权限混淆
- 业务对象定位错误

### 哈希输入域隔离不清

如果不同用途复用相同拼接规则，可能导致语义冲突。

## 10. 一句话记忆

Solana 密码学最常见的三根主线是：

- `Ed25519` 负责身份与签名
- `SHA-256` 负责哈希与时间序基础
- `PDA` 负责程序控制下的确定性地址与权限模型

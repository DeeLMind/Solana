# Solana 共识算法

## 1. 先说结论

Solana 的共识不是单一算法，而是一套组合：

- `Proof of Stake` 决定谁有更大投票权重。
- `Proof of History` 提供可验证时间序。
- `Tower BFT` 在这个时间轴上完成拜占庭容错共识。

因此如果有人问“Solana 用什么共识”，更准确的回答是：

- **PoS + PoH + Tower BFT**

## 2. Proof of Stake

在 Solana 中，验证者的投票权重主要来自质押的 stake。

stake 越高，意味着：

- 更容易在领导者排班中获得出块机会。
- 投票权重更高。
- 对网络 finality 的影响更大。

这里的核心点不是“谁算力强谁赢”，而是“谁被委托的 stake 更高，谁在共识中的权重更高”。

## 3. Proof of History

PoH 可以理解为一个可验证时钟。

它并不是直接裁决区块正确性的终局规则，而是解决一个很现实的问题：

- 分布式网络里，节点彼此通信有延迟。
- 如果每一步都要先协调“现在几点了、谁先谁后”，成本很高。
- PoH 通过连续哈希链给出一个难以伪造的事件顺序参考。

### PoH 的直觉理解

可以把 PoH 想成：

- 一个持续向前推进的哈希时钟
- 每个 tick 都建立在前一个结果之上
- 某个事件插入在哪个位置，别人以后可以验证

这让网络在排序问题上减少了很多额外通信。

## 4. Tower BFT

Tower BFT 是 Solana 在 PoH 时钟上运行的拜占庭容错投票机制。

它的核心思想是：

- 验证者对某个 fork 上的 slot 投票。
- 投票会带来 lockout。
- 连续投票会让 lockout 指数增长。
- 如果验证者想轻易改投另一个分叉，就会破坏自己之前的锁定承诺。

这让系统逐步收敛到某条主链。

## 5. 什么是 lockout

lockout 是 Tower BFT 的关键概念。

直觉上它表示：

- 你投了某个分叉之后，不是马上就能无成本反悔。
- 你越往后持续支持这个分叉，承诺越深。
- 承诺越深，切换到别的分叉代价越大。

这会减少频繁反复横跳，帮助网络尽快形成稳定共识。

## 6. Slot、Epoch、Leader Schedule

### Slot

slot 可以理解为出块时间片。每个 slot 通常对应一个 leader 的出块机会。

### Epoch

epoch 是更大的时间周期，很多与 staking、leader schedule 相关的参数会按 epoch 更新。

### Leader Schedule

leader schedule 表示在未来一段时间内，哪些验证者将在什么 slot 成为 leader。

它的意义在于：

- 验证者可以提前知道自己何时出块。
- 客户端和网络可以提前把交易转发给未来 leader。
- Gulf Stream 因此能够工作得更高效。

## 7. Finality 是怎么形成的

Solana 的确认通常不是“只有一种确认”。

常见可以粗略理解成：

- `processed`：节点已经处理了交易。
- `confirmed`：交易所在区块已获得较强投票确认。
- `finalized`：经过更深层投票锁定，回滚概率极低。

最终性来自：

- 验证者不断在后续 slot 上投票
- Tower BFT 的 lockout 深化
- 分叉逐渐被淘汰

## 8. Solana 共识为什么快

共识快并不只是因为“出块间隔短”，更关键是：

- PoH 把时间顺序问题提前处理。
- leader schedule 可提前确定。
- 交易可以预先路由到未来 leader。
- Tower BFT 在已有时序上做投票，而不是每轮都重新协调大量元信息。

## 9. Solana 共识的代价和挑战

高性能不是没有代价，常见挑战包括：

- 对网络稳定性和节点硬件要求较高。
- 分叉恢复与重放逻辑复杂。
- 高吞吐下，验证者软件实现和网络传播机制压力很大。
- 节点实现中的工程问题会比很多慢链更容易放大。

## 10. 一个记忆公式

记住这句就够了：

- `PoS` 决定权重
- `PoH` 提供时间轴
- `Tower BFT` 在时间轴上投票并形成 finality

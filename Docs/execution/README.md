# Solana 交易与执行模型

## 1. 执行模型为什么重要

如果账户模型解释了“状态放哪”，那么执行模型解释的就是：

- 交易怎么进入系统
- 指令怎么调用程序
- 程序怎么读写账户
- 多个程序怎么互相调用

## 2. Transaction 和 Instruction

在 Solana 中，一笔 `transaction` 里可以包含多条 `instruction`。

可以这样理解：

- transaction 是外层容器
- instruction 是具体调用动作

一笔交易可能会：

- 先创建账户
- 再初始化账户
- 再执行业务逻辑

这些都能组合在同一笔 transaction 里。

## 3. Account Metas

每条 instruction 都会携带相关账户元信息，常见包括：

- 哪些账户要传入
- 哪些账户可写
- 哪些账户必须签名

这一步非常关键，因为运行时依赖这些声明去做：

- 权限验证
- 账户锁定
- 并行执行判断

## 4. SVM

SVM 是 Solana 的执行环境。

它负责：

- 加载程序
- 提供 syscall
- 维护调用上下文
- 执行 CPI
- 计量 compute units
- 处理账户读写规则

所以开发者写的程序，真正不是“直接在链上裸跑”，而是运行在 SVM 这个环境中。

## 5. Sealevel

Sealevel 是并行执行层。

它根据交易声明的账户集合判断是否冲突：

- 读读通常可并行
- 读写、写写通常会冲突

这和传统 EVM 链常见的全局串行执行有本质区别。

## 6. CPI

`CPI` 是 Cross Program Invocation，也就是程序调用程序。

常见场景：

- 你的程序调用 System Program 创建账户
- 调用 Token Program 转账
- 调用关联账户程序创建 ATA

这和传统语言里的函数调用很像，但安全边界复杂得多，因为：

- 被调用方可能修改你传入的账户
- 权限和 signer 传播要严格理解
- 返回后不能盲目信任上下文没变

## 7. Compute Units

Solana 不直接沿用 EVM 的 gas 表达，但有自己的资源计量：

- `compute units`

每条指令和系统调用都会消耗一定 CU。其意义是：

- 防止程序无限消耗资源
- 给交易定价提供基础
- 帮助调度器限制单笔交易成本

## 8. Program Execution 基本路径

一条 instruction 大致会经过：

1. 校验交易与签名
2. 校验账户元信息
3. 加载相关账户
4. 加载目标程序
5. 在 SVM 中执行程序逻辑
6. 更新账户状态
7. 返回结果并记录日志

## 9. 原子性

Solana transaction 默认具有原子性。

这意味着：

- 如果交易中某一步失败
- 整笔交易通常都会失败回滚

这让开发者可以把多个操作安全地打包到一笔交易里。

## 10. 日志与模拟

开发时非常重要的两个工具是：

- `simulateTransaction`
- 程序日志

因为 Solana 程序执行环境比较严格，很多问题只有在真实运行时上下文里才容易暴露，例如：

- 账户权限错误
- CPI 调用失败
- CU 超限
- PDA 派生错误

## 11. 开发和审计最常见的执行层风险

- 未校验账户身份
- CPI 后状态假设错误
- 重入语义理解不清
- 权限传播理解错误
- compute budget 边界没考虑
- 并行环境下错误假设执行顺序

## 12. 一句话记忆

Solana 的执行模型可以概括为：

- **交易携带账户声明，SVM 执行指令，Sealevel 尽量并行化，CPI 连接程序生态。**
